== Overview

BRF Besiktningsschema is a single-page application for Swedish housing cooperative (BRF) administrators to create apartment inspection schedules. The administrator configures a building's structure, generates an inspection schedule across a date range, writes per-apartment notification letters using templates with placeholders, and prints either individual letters or a schedule overview via the browser's native print dialog.

The application is a static site compiled to vanilla HTML, CSS, and JavaScript. It has zero runtime dependencies. All user data is persisted in the browser's localStorage. There is no backend, no authentication, and no external services.

=== Models (Business Logic)

Pure TypeScript functions with no side effects. Each feature has a type definition file and a model file:

[cols="1,2", options="header"]
|===
| Module | Responsibility

| `building.ts` / `building-model.ts`
| Building configuration types. Validation (floors 1-30, apartments 1-20), configurable apartment numbering start (1000-9999), flexible digit position assignment for floor and apartment numbers within the 4-digit ID, apartment ID generation via digit substitution algorithm, floor adjustment, default config.

| `schedule.ts` / `schedule-model.ts`
| Schedule types. Date arithmetic, ISO date handling (`YYYY-MM-DD`), 24-hour time formatting (`HH:MM` via minutes since midnight), weekend/exclusion filtering, lunch break handling, scheduling algorithm (sequential floor-by-floor assignment with manual overrides), configurable print access labels, per-apartment access selections, validation.

| `template.ts` / `template-model.ts`
| Template types. Placeholder parsing (`{...}` tokens), data field resolution (apartment + appointment data), template rendering (string substitution), default placeholder mappings where `{datum}` uses ISO date field while legacy Swedish date mapping remains supported.

| `print.ts` / `print-model.ts`
| Print data types. Composes data from all three upstream features: generates per-apartment letter data (rendered templates) and schedule overview data (date-grouped appointment tables with ISO date headings, 24-hour times, and a configurable tenant-access column).
|===

=== Services (Persistence)

A single module `storage.ts` provides save/load/clear functions for each data domain:

[cols="1,1", options="header"]
|===
| localStorage Key | Data

| `brf-schedule:building`
| `BuildingConfig` as JSON

| `brf-schedule:schedule`
| `ScheduleConfig`, `ManualOverride[]`, and per-apartment access selections as JSON

| `brf-schedule:templates`
| `MessageTemplate[]` as JSON
|===

All operations are wrapped in `try/catch` for robustness (private browsing, quota exceeded). The print feature has no persistence.

=== Components (UI)

13 Svelte 5 components using runes (`$state`, `$derived`, `$effect`) for reactivity:

[cols="1,1,2", options="header"]
|===
| Feature | Components | Role

| Building Configuration
| `BuildingConfigPanel`, `FloorInputRow`, `BuildingFacade`
| Floor/apartment inputs, SVG facade illustration, validation display

| Inspection Scheduling
| `ScheduleConfigPanel`, `ScheduleView`, `ScheduleAppointmentRow`
| Date/time/exclusion configuration, access-label configuration, appointment list, manual overrides, per-appointment access selection

| Message Templates
| `MessageTemplatePanel`, `TemplateList`, `TemplateEditor`, `TemplatePreview`
| Template CRUD, placeholder mapping, live preview with real data

| Print Output
| `PrintPanel`, `PrintLetterPage`, `PrintScheduleOverview`
| Mode selection, A4 letter rendering, schedule table with access column, print trigger
|===

`App.svelte` is the root component and data flow hub. It loads persisted data on startup, computes shared derived state (apartments, schedule result), and passes data down to feature panels via props and callbacks.

== Data Flow

Data flows through the four features in a pipeline, with each feature consuming output from the previous ones:

[source,mermaid]
----
flowchart LR
  subgraph Input ["User Input"]
    U1["Building structure<br/>(floors, apartments)"]
    U2["Schedule parameters<br/>(dates, times, exclusions)"]
    U3["Template text<br/>(body, placeholders)"]
    U4["Print mode selection"]
  end

  subgraph Processing ["Business Logic"]
    BM["building-model<br/>→ Apartment[]"]
    SM["schedule-model<br/>→ ScheduleResult"]
    TEM["template-model<br/>→ rendered text"]
    PM["print-model<br/>→ print data"]
  end

  subgraph Output ["Output"]
    SVG["SVG facade illustration"]
    LIST["Appointment list"]
    PREVIEW["Template preview"]
    PRINT["A4 print / PDF"]
  end

  U1 --> BM --> SM
  U2 --> SM --> TEM
  U3 --> TEM --> PM
  U4 --> PM
  BM --> SVG
  SM --> LIST
  TEM --> PREVIEW
  PM --> PRINT
----

1. **Building Configuration** produces `Apartment[]` from the user-defined building structure.
2. **Inspection Scheduling** consumes `Apartment[]` and schedule parameters to produce `ScheduleResult` (appointments assigned to dates/times).
3. **Message Templates** consumes `Apartment[]` and `ScheduleResult` to resolve placeholders in user-written templates into rendered text.
4. **Print Output** consumes all three upstream outputs to compose A4-formatted individual letters or a schedule overview table, triggering `window.print()`.

All intermediate state is reactive. Changes to building configuration cascade through the schedule, templates, and print output automatically via Svelte 5 `$derived` runes in `App.svelte`.

== Deployment

[source,mermaid]
----
flowchart LR
  DEV["Developer"] -->|"git push"| GH["GitHub Repository"]
  GH -->|"GitHub Actions<br/>vite build"| PAGES["GitHub Pages<br/>(static files)"]
  PAGES -->|"HTTPS"| BROWSER["User's Browser"]
  BROWSER -->|"localStorage API"| LS["localStorage<br/>(user data)"]
----

The application is deployed as static files on GitHub Pages:

* GitHub Actions workflow `.github/workflows/deploy-pages.yml` runs on pushes to `main` and manual dispatch.
* The workflow runs `npm ci`, `npm test`, and `npm run build`, then uploads `dist/` via `actions/upload-pages-artifact` and deploys with `actions/deploy-pages`.
* Build uses `BASE_PATH=/<repo>/` so Vite emits correct asset URLs for GitHub Pages project-site hosting.
* No server-side code, no API endpoints, no backend infrastructure.
* User data lives entirely in the browser's localStorage and never leaves the user's device.
