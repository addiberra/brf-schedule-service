== Technical Context

This feature is already implemented and stays behaviorally unchanged. This design revision updates *how* the UI is implemented to align with the accepted architecture migration and redesign direction:

* Tailwind CSS v4 utility-first styling replaces large component-scoped CSS.
* Bits UI primitives become the standard interaction layer for key controls.
* The application shell shifts from a single long page to tabbed top-level navigation.
* The building facade keeps the same functional outputs while being visually redesigned for a warm professional look.

All requirements in `spec.adoc` remain the source of truth and are still satisfied without adding or removing feature behavior.

== Decisions and Constraints

* <<../../adrs/0001-svelte5-vite6-typescript-static-site.adoc#,ADR-0001>>: Svelte 5 + Vite 6 + TypeScript remain the base stack.
* <<../../adrs/0002-browser-localstorage-persistence.adoc#,ADR-0002>>: Building configuration persistence remains browser `localStorage` under `brf-schedule:*` keys.
* <<../../adrs/0003-tailwindcss-v4-bits-ui-warm-tabbed-redesign.adoc#,ADR-0003>>: Tailwind CSS v4 + Bits UI and tabbed navigation are the accepted architecture/tooling direction for redesign.

From `docs/constraints.adoc`:

* Build output remains static HTML/CSS/JS for GitHub Pages.
* No server-side code or external services are introduced.
* Data remains exclusively in `localStorage`.
* UI text remains Swedish.
* TypeScript strict mode and test quality constraints remain in force.

== Research and Unknowns

No unresolved unknowns remain for this design update.

Resolved research points:

* Tailwind v4 utility composition can express the full warm redesign without changing business logic modules.
* Bits UI primitives (Tabs, Checkbox, Button, Label, Dialog) satisfy interaction/accessibility needs while preserving existing behavior.
* The redesigned facade remains SVG-based and reactive; no canvas or external graphics library is required.

=== UI Architecture and Styling Strategy

==== App shell and tabbed navigation

`App.svelte` is restructured around Bits UI Tabs as the top-level navigation container. Each major feature renders inside a tab panel. The building-configuration tab keeps current data flow and callbacks.

Implementation notes:

* Tabs state is UI-only and does not affect persistence schema.
* The building configuration panel mounts/unmounts via tab panels without changing persisted building data behavior.
* Keyboard and ARIA behavior for tabs are provided by Bits UI primitives.

==== Tailwind CSS v4 utility approach

Tailwind utilities are the default styling mechanism. The feature defines a warm professional visual language through reusable utility patterns and CSS variables exposed at shell level.

Design tokens (example variable set, exact values can be adjusted during implementation):

[source,css]
----
:root {
  --surface-0: #f7f4ef;

  --text-strong: #2f2a24;
  --text-muted: #5f5549;
  --accent-warm: #a96f3f;
  --accent-warm-strong: #8b552f;
  --line-soft: #d6c7b3;

----

Utility composition principles:

* Layout: `grid`, `flex`, and responsive breakpoint utilities (`sm`, `md`, `lg`) replace custom layout CSS.
* Surface styling: layered warm backgrounds, subtle borders, and soft shadows via utility classes.
* Typography: utility-defined scale/weight for hierarchy; Swedish labels remain unchanged in content.
* State styling: validation, hover, focus-visible, and disabled states are expressed with utility variants.

==== Bits UI primitives usage in this feature

* `Checkbox` + `Label`: uniform apartment mode toggle (US-007).
* `Button`: reset trigger and other panel actions.
* `Dialog`: reset confirmation flow replacing browser-native confirm while keeping same decision behavior.
* Tabs remain in app shell and affect access to this panel contextually.

=== BuildingConfigPanel responsibilities

The panel remains the orchestration component for this feature. Existing domain behavior is preserved; only interaction composition and visual layer change.

State and derivations:

* `config` from `loadBuilding()` fallback to `createDefaultConfig()`.
* `validationErrors` keyed by field id.
* `uniformMode` as non-persisted UI state.
* `apartments` and `totalCount` as reactive derivations from model functions.
* Autosave effect remains gated by valid state and writes with `saveBuilding(config)`.

Input rendering strategy:

* Floor count and numbering start inputs are rendered in a responsive form card.
* Uniform mode ON: per-floor rows are hidden and replaced by one shared apartment count input.
* Uniform mode OFF: per-floor rows return with current values preserved.
* Validation messages are inline and associated with controls.

Reset flow:

* Reset action opens Bits Dialog.
* Confirm path: clear persisted state and restore default config.
* Cancel path: no mutation.

=== BuildingFacade redesign approach

`BuildingFacade.svelte` keeps SVG as rendering technology and keeps all functional mapping (floors, apartments, labels, real-time updates). The redesign focuses on aesthetics and responsive framing.

Facade principles:

* Warm professional palette with gradient walls and atmospheric background.
* Clear apartment delineation with frames, headers, sills, and readable labels.
* Architectural details (roof ridge, chimney, door, separators, alternating balconies, ground vegetation) remain dynamically computed from configuration.
* SVG scales responsively inside a Tailwind-styled container (`w-full`, bounded max width, preserved aspect ratio).

Responsive behavior:

* Mobile: facade appears below controls, reduced outer spacing, labels remain readable via minimum text size strategy.
* Tablet/Desktop: two-column layout (controls + facade) where width permits.
* Large apartment counts: horizontal overflow is avoided by viewBox scaling rather than clipping.

=== Data Model and Persistence

==== BuildingConfig extensions

The `BuildingConfig` interface is extended with two new optional fields to support flexible digit position assignment:

[source,typescript]
----
export interface BuildingConfig {
  floorCount: number;
  floors: FloorConfig[];
  apartmentNumberStart: number;      // Range: 1000-9999 (expanded from 1001-1901)
  levelDigits?: [number, number];    // [startPos, endPos], 1-indexed, e.g. [1,1] or [1,2]
  apartmentDigits?: [number, number]; // Same format, must not overlap with levelDigits
}
----

**Digit position tuples:**

* Each tuple represents a contiguous range of digit positions in a 4-digit apartment number.
* Position 1 is the leftmost digit, position 4 is the rightmost.
* Single digit: `[1, 1]`, `[2, 2]`, `[3, 3]`, or `[4, 4]`
* Consecutive pairs: `[1, 2]`, `[2, 3]`, or `[3, 4]`
* The ranges must not overlap -- `levelDigits` and `apartmentDigits` must cover disjoint positions.

**Backward compatibility defaults:**

When `levelDigits` or `apartmentDigits` are undefined (legacy data), the system applies:

* `levelDigits: [1, 1]` -- first digit represents the floor
* `apartmentDigits: [3, 4]` -- last two digits represent the apartment position
* `apartmentNumberStart: 1001` -- default start value

These defaults produce identical output to the previous algorithm for existing configurations.

==== Apartment numbering start range update

The `apartmentNumberStart` range is expanded:

* **New range:** 1000-9999 (any 4-digit value)
* **Previous range:** 1001-1901 in steps of 100
* **Step restriction:** Removed -- any integer in range is valid

==== Legacy data migration

In `storage.ts`, when loading saved configuration:

1. If `levelDigits` is missing, set to `[1, 1]`
2. If `apartmentDigits` is missing, set to `[3, 4]`
3. If `apartmentNumberStart` is missing, set to `1001`
4. If `apartmentNumberStart` is present but uses old step-100 values, preserve as-is (still valid in new range)

This ensures seamless migration for existing users.

==== Apartment identifier generation algorithm

The `generateApartmentId` function is updated to support flexible digit positions:

[source,typescript]
----
function generateApartmentId(
  floor: number,
  position: number,
  apartmentNumberStart: number,
  levelDigits: [number, number],
  apartmentDigits: [number, number]
): string {
  // 1. Convert start number to 4-character array (zero-pad if needed)
  const startStr = apartmentNumberStart.toString().padStart(4, '0');
  const result = startStr.split('');

  // 2. Calculate digit counts for each range
  const levelDigitCount = levelDigits[1] - levelDigits[0] + 1;
  const apartmentDigitCount = apartmentDigits[1] - apartmentDigits[0] + 1;

  // 3. Format floor number with leading zeros to fit level digit range
  const floorStr = floor.toString().padStart(levelDigitCount, '0');

  // 4. Substitute floor digits into result at levelDigits positions (1-indexed)
  for (let i = 0; i < levelDigitCount; i++) {
    result[levelDigits[0] - 1 + i] = floorStr[i];
  }

  // 5. Format apartment position with leading zeros to fit apartment digit range
  const posStr = position.toString().padStart(apartmentDigitCount, '0');

  // 6. Substitute apartment digits into result at apartmentDigits positions (1-indexed)
  for (let i = 0; i < apartmentDigitCount; i++) {
    result[apartmentDigits[0] - 1 + i] = posStr[i];
  }

  // 7. Return result as string
  return result.join('');
}
----

**Algorithm examples:**

|===
| Start | Level Digits | Apt Digits | Floor 2, Apt 3 | Explanation

| 1001 | [1, 1] | [3, 4] | 2003 | Floor in pos 1, apt in pos 3-4, "00" from start preserved in pos 2
| 1234 | [2, 2] | [4, 4] | 1234 → 1233 | Floor in pos 2, apt in pos 4, "1" and "3" fixed
| 5000 | [1, 2] | [3, 4] | 0203 | Floor 02 in pos 1-2, apt 03 in pos 3-4
| 1100 | [3, 3] | [4, 4] | 1123 | Floor in pos 3, apt in pos 4, "11" fixed
|===

==== Validation functions

New validation functions in `building-model.ts`:

**validateDigitPositions:**
[source,typescript]
----
function validateDigitPositions(
  levelDigits: [number, number],
  apartmentDigits: [number, number]
): ValidationResult {
  // Ranges must be consecutive (endPos >= startPos)
  if (levelDigits[1] < levelDigits[0] || apartmentDigits[1] < apartmentDigits[0]) {
    return { valid: false, error: 'Ogiltigt positionsintervall.' };
  }

  // Ranges must not overlap
  const levelRange = new Set<number>();
  for (let i = levelDigits[0]; i <= levelDigits[1]; i++) levelRange.add(i);

  for (let i = apartmentDigits[0]; i <= apartmentDigits[1]; i++) {
    if (levelRange.has(i)) {
      return { valid: false, error: 'Positioner för våning och lägenhet får inte överlappa.' };
    }
  }

  return { valid: true };
}
----

**validateDigitCapacity:**
[source,typescript]
----
function validateDigitCapacity(
  digits: [number, number],
  maxValue: number,
  fieldLabel: string
): ValidationResult {
  const digitCount = digits[1] - digits[0] + 1;
  const maxCapacity = digitCount === 1 ? 9 : 99; // single digit: 1-9, two digits: 1-99

  if (maxValue > maxCapacity) {
    return {
      valid: false,
      error: `${fieldLabel} (${maxValue}) överstiger kapaciteten för ${digitCount} siffra (max ${maxCapacity}).`,
    };
  }

  return { valid: true };
}
----

**Updated validateApartmentNumberStart:**
[source,typescript]
----
function validateApartmentNumberStart(value: number): ValidationResult {
  if (!Number.isInteger(value)) {
    return { valid: false, error: 'Startnummer måste vara ett heltal.' };
  }

  if (value < 1000 || value > 9999) {
    return { valid: false, error: 'Startnummer måste vara mellan 1000 och 9999.' };
  }

  return { valid: true };
}
----

=== Digit Position UI Controls

==== Control layout in BuildingConfigPanel

The configuration form is extended with two dropdown controls for digit position selection, placed after the numbering start input:

**Floor digit position dropdown ("Siffra för våning"):**

* Label: "Siffra för våning"
* Options:
** "Position 1" → `[1, 1]`
** "Position 2" → `[2, 2]`
** "Position 3" → `[3, 3]`
** "Position 4" → `[4, 4]`
** "Positioner 1-2" → `[1, 2]`
** "Positioner 2-3" → `[2, 3]`
** "Positioner 3-4" → `[3, 4]`
* Default: "Position 1" (`[1, 1]`)

**Apartment digit position dropdown ("Siffra för lägenhet"):**

* Label: "Siffra för lägenhet"
* Same option set as floor dropdown
* Default: "Positioner 3-4" (`[3, 4]`)
* When floor digit selection changes, apartment options that would overlap are shown with an error state if selected

**Numbering start input update:**

* Input attributes changed: `min=1000 max=9999 step=1`
* Removes the step-100 restriction from the previous design

==== Live preview display

A preview component displays below the digit position controls:

* Format: "Våning N, Lgh M → XXXX"
* Example values: Floor 2, Apartment 3 (fixed example values for consistent preview)
* The computed value uses the current `apartmentNumberStart`, `levelDigits`, and `apartmentDigits`
* Updates immediately when any of the three inputs change

Preview implementation:

[source,svelte]
----
<p class="text-sm text-muted">
  Våning 2, Lgh 3 → {generateApartmentId(2, 3, config.apartmentNumberStart, config.levelDigits, config.apartmentDigits)}
</p>
----

==== Validation feedback

Validation errors are displayed inline:

* Overlapping digit positions: error message below both dropdowns
* Floor digit capacity exceeded: error message indicating floor count exceeds digit range
* Apartment digit capacity exceeded: error message indicating max apartments exceeds digit range

All validation errors block the save path (same pattern as existing validations).
