= Specification: Building Configuration
:version: 1.2.0
:source-stories-version: 1.3.0
:last-updated: 2026-03-03
:feature: building-configuration
:prefix: BCFG
:toc:

== BCFG-001
The system shall provide an input field for the number of floors in the building.

Type:: Ubiquitous
Source:: US-001: Configure Number of Floors, Criterion 1
Verification:: Open the building configuration view. Confirm that an input field for the number of floors is visible.

== BCFG-002
The system shall accept a floor count between 1 and 30 inclusive.

Type:: Ubiquitous
Source:: US-001: Configure Number of Floors, Criterion 2
Verification:: Enter floor counts of 1, 15, and 30. Confirm all are accepted without error.

== BCFG-003
If the user enters a floor count less than 1 or greater than 30, then the system shall display a validation error and prevent saving.

Type:: Unwanted behavior
Source:: US-001: Configure Number of Floors, Criterion 3
Verification:: Enter 0 as the floor count. Confirm a validation error is displayed and the value is not saved. Enter 31. Confirm the same behavior.

== BCFG-004
When the user changes the floor count, the system shall update the building illustration to reflect the new number of floors.

Type:: Event-driven
Source:: US-001: Configure Number of Floors, Criterion 4
Verification:: Set the floor count to 3. Confirm the building illustration shows 3 floors. Change to 5. Confirm the illustration updates to show 5 floors.

== BCFG-005
The system shall provide an input field for the number of apartments on each floor.

Type:: Ubiquitous
Source:: US-002: Configure Apartments per Floor, Criterion 1
Verification:: Configure a building with 3 floors. Confirm that each floor has its own input field for the number of apartments.

== BCFG-006
The system shall allow a different number of apartments per floor.

Type:: Ubiquitous
Source:: US-002: Configure Apartments per Floor, Criterion 2
Verification:: Configure a 3-floor building. Set floor 1 to 2 apartments, floor 2 to 3 apartments, floor 3 to 4 apartments. Confirm all values are accepted and reflected independently.

== BCFG-007
The system shall accept an apartment count between 1 and 20 per floor.

Type:: Ubiquitous
Source:: US-002: Configure Apartments per Floor, Criterion 3
Verification:: Enter apartment counts of 1, 10, and 20 for a floor. Confirm all are accepted without error.

== BCFG-008
If the user enters an apartment count less than 1 or greater than 20 for any floor, then the system shall display a validation error for that floor.

Type:: Unwanted behavior
Source:: US-002: Configure Apartments per Floor, Criterion 4
Verification:: Enter 0 as the apartment count for floor 2. Confirm a validation error is displayed for floor 2 specifically. Enter 21. Confirm the same behavior. Confirm other floors are unaffected.

== BCFG-009
When the user changes the apartment count for a floor, the system shall update the building illustration for that floor.

Type:: Event-driven
Source:: US-002: Configure Apartments per Floor, Criterion 5
Verification:: Set floor 1 to 2 apartments. Confirm the illustration shows 2 apartment elements on floor 1. Change floor 1 to 4 apartments. Confirm the illustration updates to show 4 elements on floor 1.

== BCFG-010
The system shall display the total number of apartments across all floors.

Type:: Ubiquitous
Source:: US-002: Configure Apartments per Floor, Criterion 6
Verification:: Configure a 2-floor building with 3 apartments on floor 1 and 4 apartments on floor 2. Confirm the displayed total is 7.

== BCFG-011
The system shall automatically assign a unique identifier to each apartment based on floor and position.

Type:: Ubiquitous
Source:: US-003: Apartment Numbering, Criterion 1
Verification:: Configure a building with 2 floors and 2 apartments per floor. Confirm each apartment has a distinct identifier.

== BCFG-012
The system shall display apartment identifiers in a format that combines floor number and apartment position (e.g., "1001", "1002" for floor 1, apartments 1 and 2).

Type:: Ubiquitous
Source:: US-003: Apartment Numbering, Criterion 2
Verification:: Configure a building with floor 1 having 2 apartments. Confirm identifiers are displayed as "1001" and "1002". Configure floor 2 with 3 apartments. Confirm identifiers are "2001", "2002", and "2003".

== BCFG-013
When the building configuration changes, the system shall recalculate all apartment identifiers.

Type:: Event-driven
Source:: US-003: Apartment Numbering, Criterion 3
Verification:: Configure a 2-floor building with 2 apartments each. Note the identifiers. Change floor 1 to 3 apartments. Confirm all identifiers are recalculated to reflect the new configuration.

== BCFG-014
The system shall display a front-facade illustration of the building.

Type:: Ubiquitous
Source:: US-004: Building Facade Illustration, Criterion 1
Verification:: Open the building configuration view with a valid configuration. Confirm a front-facade illustration of the building is visible.

== BCFG-015
The system shall generate the illustration dynamically based on the configured number of floors and apartments per floor.

Type:: Ubiquitous
Source:: US-004: Building Facade Illustration, Criterion 2
Verification:: Configure a 2-floor building with 3 apartments on each floor. Confirm the illustration shows 2 floors with 3 apartment elements each. Change to 4 floors with 2 apartments each. Confirm the illustration updates accordingly.

== BCFG-016
The system shall display each apartment as a distinct visual element on the facade.

Type:: Ubiquitous
Source:: US-004: Building Facade Illustration, Criterion 3
Verification:: Configure a building with 2 floors and 3 apartments per floor. Confirm that 6 distinct apartment elements are visible in the illustration.

== BCFG-017
The system shall label each apartment element in the illustration with its apartment identifier.

Type:: Ubiquitous
Source:: US-004: Building Facade Illustration, Criterion 4
Verification:: Configure a building with 1 floor and 2 apartments. Confirm that each apartment element in the illustration displays its identifier ("1001", "1002").

== BCFG-018
When the building configuration changes, the system shall update the illustration in real time.

Type:: Event-driven
Source:: US-004: Building Facade Illustration, Criterion 5
Verification:: Change the floor count or apartment count. Confirm the illustration updates immediately without a page reload or explicit save action.

== BCFG-019
When the user modifies the building configuration, the system shall persist the configuration to browser storage.

Type:: Event-driven
Source:: US-005: Save Building Configuration, Criterion 1
Verification:: Modify the floor count. Close the browser tab. Reopen the application. Confirm the modified floor count is still present.

NOTE: The storage mechanism (localStorage with `brf-schedule:` namespace) is enforced by project constraint (Data: "All user data shall be stored exclusively in the browser's localStorage API" and "localStorage keys shall be namespaced"). Not duplicated here.

== BCFG-020
When the application loads, the system shall restore the previously saved building configuration.

Type:: Event-driven
Source:: US-005: Save Building Configuration, Criterion 2
Verification:: Configure a building with 5 floors and varying apartments per floor. Close and reopen the application. Confirm the same 5-floor configuration with all apartment counts is displayed.

== BCFG-021
If no saved configuration exists, then the system shall display a default configuration state.

Type:: Unwanted behavior
Source:: US-005: Save Building Configuration, Criterion 3
Verification:: Clear all application data from browser storage. Open the application. Confirm the system displays a default configuration (not an error state).

== BCFG-022
The system shall provide a reset button for the building configuration.

Type:: Ubiquitous
Source:: US-006: Reset Building Configuration, Criterion 1
Verification:: Open the building configuration view. Confirm a reset button is visible.

== BCFG-023
When the user activates the reset function, the system shall ask for confirmation before proceeding.

Type:: Event-driven
Source:: US-006: Reset Building Configuration, Criterion 2
Verification:: Click the reset button. Confirm a confirmation prompt is displayed before any changes are made.

== BCFG-024
When the user confirms the reset, the system shall clear the saved configuration and restore default values.

Type:: Event-driven
Source:: US-006: Reset Building Configuration, Criterion 3
Verification:: Configure a building with 10 floors. Click reset. Confirm the reset. Confirm the configuration returns to default values and the saved data is cleared.

== BCFG-025
If the user cancels the reset, then the system shall leave the configuration unchanged.

Type:: Unwanted behavior
Source:: US-006: Reset Building Configuration, Criterion 4
Verification:: Configure a building with 10 floors. Click reset. Cancel the reset. Confirm the configuration remains at 10 floors with all previous values intact.

== BCFG-026
The system shall provide a toggle control to enable or disable uniform apartment count mode.

Type:: Ubiquitous
Source:: US-007: Uniform Apartment Count Per Level, Criterion 1
Verification:: Open the building configuration view. Confirm a labeled toggle (checkbox) for uniform apartment count mode is visible below the floor count input.

== BCFG-027
When uniform mode is enabled, the system shall replace the per-floor apartment inputs with a single shared input field.

Type:: Event-driven
Source:: US-007: Uniform Apartment Count Per Level, Criterion 2
Verification:: Enable uniform mode. Confirm the per-floor input rows disappear and a single shared apartment count input is shown.

== BCFG-028
When the user changes the shared uniform input value, the system shall apply the same apartment count to all floors.

Type:: Event-driven
Source:: US-007: Uniform Apartment Count Per Level, Criterion 3
Verification:: Enable uniform mode. Set the shared input to 4. Confirm all floors in the underlying configuration have 4 apartments. Confirm the facade illustration reflects 4 apartments on every floor.

== BCFG-029
The system shall accept a uniform apartment count between 1 and 20.

Type:: Ubiquitous
Source:: US-007: Uniform Apartment Count Per Level, Criterion 4
Verification:: Enable uniform mode. Enter 1, 10, and 20. Confirm all values are accepted without error.

== BCFG-030
If the user enters a uniform apartment count less than 1 or greater than 20, then the system shall display a validation error.

Type:: Unwanted behavior
Source:: US-007: Uniform Apartment Count Per Level, Criterion 5
Verification:: Enable uniform mode. Enter 0. Confirm a validation error is shown. Enter 21. Confirm the same behavior.

== BCFG-031
When uniform mode is disabled, the system shall show per-floor inputs with the current apartment counts preserved.

Type:: Event-driven
Source:: US-007: Uniform Apartment Count Per Level, Criterion 6
Verification:: Set 3 floors. Enable uniform mode. Set count to 5. Disable uniform mode. Confirm per-floor inputs appear, each showing 5 apartments. Modify floor 2 to 3 apartments. Confirm the change is independent per floor.

== BCFG-032
When the user enables uniform mode, the system shall set all floors to the apartment count of floor 1.

Type:: Event-driven
Source:: US-007: Uniform Apartment Count Per Level, Criterion 7
Verification:: Set floor 1 to 3, floor 2 to 5, floor 3 to 7. Enable uniform mode. Confirm the shared input shows 3 and all floors now have 3 apartments.

== BCFG-033
The system shall render the building facade with gradient or textured walls that convey material depth.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 1
Verification:: Open the building configuration view. Confirm the building walls in the SVG illustration use a visible gradient or texture fill rather than a flat solid color.

== BCFG-034
The system shall display window frames with sills and headers on each apartment element in the facade.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 2
Verification:: Configure a building with apartments. Confirm each apartment window in the facade has a visible sill (below the window) and header (above the window) in addition to the window frame.

== BCFG-035
The system shall display a detailed roof with a ridge line and chimney.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 3
Verification:: Confirm the roof polygon has a visible ridge detail and that a chimney shape is rendered on one side of the roof.

== BCFG-036
The system shall display an entrance door centered on the ground floor of the facade.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 4
Verification:: Configure a building with at least 1 floor. Confirm a door shape is visible at the base of the ground floor, centered horizontally.

== BCFG-037
The system shall display balconies on alternating apartment elements in the facade.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 5
Verification:: Configure a building with 3 floors and 4 apartments per floor. Confirm that balcony elements appear on every other apartment, creating a staggered visual pattern.

== BCFG-038
The system shall display horizontal floor separator lines between each level in the facade.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 6
Verification:: Configure a building with 3 floors. Confirm visible horizontal lines separate each floor level in the illustration.

== BCFG-039
The system shall display a ground area with vegetation below the building in the facade.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 7
Verification:: Confirm a green ground strip with vegetation shapes (bushes or trees) is rendered below the building.

== BCFG-040
The system shall apply drop shadows to the building element in the facade for visual depth.

Type:: Ubiquitous
Source:: US-008: Enhanced Building Facade Illustration, Criterion 8
Verification:: Inspect the SVG output. Confirm an SVG `<filter>` with drop shadow is defined and applied to the building shape.

== BCFG-041
When the building configuration changes, the enhanced facade shall update in real time.

Type:: Event-driven
Source:: US-008: Enhanced Building Facade Illustration, Criterion 9
Verification:: Change the floor count or apartment count. Confirm the enhanced facade (including all detail elements) updates immediately without page reload.

== BCFG-042
The system shall provide an input for apartment numbering start value for floor 1 apartment 1.

Type:: Ubiquitous
Source:: US-009: Configurable Apartment Numbering Start, Criterion 1
Verification:: Open the building configuration view. Confirm there is an input control for selecting the starting apartment number.

== BCFG-043
The system shall accept apartment numbering start values 1001 through 1901 in increments of 100.

Type:: Ubiquitous
Source:: US-009: Configurable Apartment Numbering Start, Criterion 2
Verification:: Enter 1001, 1101, and 1901 as start values and confirm they are accepted. Enter 1050 and confirm it is rejected.

== BCFG-044
If the user enters an apartment numbering start value outside the valid set, then the system shall display a validation error and prevent saving.

Type:: Unwanted behavior
Source:: US-009: Configurable Apartment Numbering Start, Criterion 3
Verification:: Enter 999, 1950, or 2001 as start value. Confirm a validation error is shown and the invalid value is not persisted.

== BCFG-045
When the user changes the apartment numbering start value, the system shall recalculate apartment identifiers for all configured apartments.

Type:: Event-driven
Source:: US-009: Configurable Apartment Numbering Start, Criterion 4
Verification:: Configure at least 2 floors and 2 apartments per floor. Change numbering start from 1001 to 1101 and confirm all apartment identifiers update accordingly.

== BCFG-046
While a building configuration contains multiple apartments, the system shall keep apartment identifiers unique across all floors and positions after numbering recalculation.

Type:: State-driven
Source:: US-009: Configurable Apartment Numbering Start, Criterion 5
Verification:: Configure a building with multiple floors and apartments, set a valid numbering start value, and confirm no duplicate apartment identifiers are displayed or generated.

== BCFG-047
When the application loads saved building data that lacks apartment numbering start value, the system shall default the numbering start value to 1001.

Type:: Event-driven
Source:: US-009: Configurable Apartment Numbering Start, Criterion 6
Verification:: Load legacy saved building data created before numbering start existed. Confirm the application sets the numbering start to 1001.
