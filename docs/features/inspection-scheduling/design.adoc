== Technical Context

This feature is already implemented and remains behaviorally unchanged. This design revision updates *how* the UI is composed and styled under the accepted redesign direction:

* Tailwind CSS v4 utility styling replaces component-scoped CSS in scheduling views.
* Bits UI primitives become the interaction baseline for scheduling controls.
* The feature is hosted inside the new tabbed application shell.
* Existing scheduling logic, persistence behavior, and spec-visible outputs remain unchanged.

All requirements in `spec.adoc` remain authoritative and are preserved.

== Decisions and Constraints

* <<../../adrs/0001-svelte5-vite6-typescript-static-site.adoc#,ADR-0001>>: Svelte 5 + Vite 6 + TypeScript remain the base stack.
* <<../../adrs/0002-browser-localstorage-persistence.adoc#,ADR-0002>>: Schedule state remains persisted in browser `localStorage` under `brf-schedule:*` keys.
* <<../../adrs/0003-tailwindcss-v4-bits-ui-warm-tabbed-redesign.adoc#,ADR-0003>>: Tailwind CSS v4 + Bits UI + tabbed shell are the accepted redesign architecture.

From `docs/constraints.adoc`:

* Static-site output only; no backend services.
* Data remains localStorage-only.
* User-visible text remains Swedish.
* TypeScript strict mode and quality constraints remain in force.

No new ADR is required for this feature update.

== Research and Unknowns

No unresolved unknowns remain.

Resolved points:

* Bits UI primitives (`Checkbox`, `Select`, `Button`, `Dialog`, `Label`) cover all interactive scheduling controls while preserving existing behavior.
* Date and time input behavior remains on native `type="date"` and `type="time"` controls to preserve exact current value semantics (`YYYY-MM-DD`, `HH:MM`) and browser affordances.
* If a richer calendar/time popover is added later, Bits UI `Popover` + `Calendar` composition is the preferred extension path, but it is not required for current behavior parity.

=== UI Architecture and Primitive Mapping

==== Tailwind CSS v4 strategy

Scheduling panels use Tailwind utility composition for layout, states, and warm visual hierarchy:

* Shared surface tokens and spacing scales from app shell design system.
* Responsive form/list layout without custom scoped CSS blocks.
* Explicit utilities for error, hover, focus-visible, and disabled states.

==== Bits UI primitives in this feature

* `Label`: all form controls in schedule configuration and manual override editor.
* `Checkbox`: toggles for lunch enablement and weekend exclusion.
* `Select`: constrained numeric/time-derived options where select semantics are clearer than free text (for example, max-per-day presets), while preserving current accepted value ranges.
* `Button`: add/remove excluded date actions, manual override actions, recalc-related controls.
* `Dialog`: non-destructive confirmation and contextual warning details (for unscheduled apartments explanation) without changing scheduling results.

Date/time primitives:

* Current behavior remains on native inputs (`type="date"`, `type="time"`) for SCHED-002, SCHED-044, SCHED-045, SCHED-046, SCHED-047 compatibility.
* Bits-based date/time popovers are optional future enhancements and are intentionally out of scope for this non-functional redesign pass.

=== Domain Logic and Persistence

No functional domain changes are introduced:

* `schedule-model.ts` remains the source of validation and scheduling rules.
* `calculateSchedule()` ordering, exclusion handling, lunch handling, daily end time handling, override preservation, and unscheduled warnings remain unchanged.
* `storage.ts` schedule save/load/clear behavior and key namespace remain unchanged.
