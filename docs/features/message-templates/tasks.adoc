== Phase 4: Fix Template Creation Reactivity _(v2.0.0 -- Current)_

Fix the bug where the editor form does not appear after clicking "Ny mall". The root cause is incorrect Svelte 5 reactivity wiring: `editingTemplate` must be a `$derived` value, the save `$effect` must use `untrack` around the parent callback, and the TemplateEditor `$effect` must track `template.id` specifically.

* [x] TASK-010: Make editingTemplate a $derived value in MessageTemplatePanel [sequential]
+
Components:: MessageTemplatePanel.svelte
Stories:: US-001
Specs:: TMPL-024
Description:: Change `editingTemplate` from `$state` (or incorrectly structured reactive variable) to a `$derived` value computed from `templates` and `selectedTemplateId` via `templates.find(t => t.id === selectedTemplateId)`. This ensures `editingTemplate` always reflects the current templates array and that the `{#if editingTemplate}` conditional renders the editor immediately when a new template is created.

* [x] TASK-011: Use untrack around parent callback in save $effect [sequential]
+
Components:: MessageTemplatePanel.svelte
Stories:: US-001, US-006
Specs:: TMPL-024, TMPL-003
Description:: In the `$effect` that persists templates to localStorage and notifies the parent when templates change, wrap the parent callback invocation with `untrack(() => ...)`. This prevents the callback from becoming a reactive dependency that could interfere with template selection state, ensuring the save fires asynchronously without disrupting the editor display.

* [x] TASK-012: Fix TemplateEditor $effect to track template.id [sequential]
+
Components:: TemplateEditor.svelte
Stories:: US-001, US-003
Specs:: TMPL-024, TMPL-011
Description:: Update the `$effect` in TemplateEditor that syncs local editing state from the incoming `template` prop to track `template.id` specifically (not the full object reference). This avoids spurious resets of the editing state during the same session while correctly re-syncing when a different template is selected or a new template is created.

== Phase 5: Locale-Aware Date Formatting _(v2.0.0 -- Current)_

Add a new `dateFormatted` data field that uses the browser's `Intl.DateTimeFormat` API for locale-aware date rendering. Retain `dateSwedish` for backward compatibility.

* [x] TASK-013: Add formatDateLocale function to schedule-model.ts [parallel]
+
Components:: schedule-model.ts
Stories:: US-007
Specs:: TMPL-025, TMPL-028
Description:: Add a `formatDateLocale(dateStr: string): string` function to `src/lib/models/schedule-model.ts`. The function parses the `YYYY-MM-DD` date string and formats it using `new Intl.DateTimeFormat(undefined, { weekday: 'long', day: 'numeric', month: 'long', year: 'numeric' }).format(date)`. Wrap the call in a try-catch; on failure, fall back to `formatDateSwedish(dateStr)` for Swedish output.

* [x] TASK-014: Add dateFormatted to DataField union and update template engine [parallel]
+
Components:: template.ts, template-model.ts
Stories:: US-007
Specs:: TMPL-025, TMPL-026, TMPL-027
Description:: Add `'dateFormatted'` to the `DataField` union type in `src/lib/models/template.ts`. Update `resolveDataField()` in `template-model.ts` to handle `'dateFormatted'` by calling `formatDateLocale()` from `schedule-model.ts`. Keep the existing `'dateSwedish'` case unchanged (calls `formatDateSwedish()`) for backward compatibility. Update `getAvailableDataFields()` to include `{ field: 'dateFormatted', label: 'Datum (formaterat)' }`.
