== Technical Context

This is an existing feature in a non-greenfield system. Building configuration and inspection scheduling are already implemented and continue to provide the apartment and appointment data used for template placeholder resolution.

The project remains a Svelte 5 + Vite 6 + TypeScript static site with browser-only persistence (`localStorage` under `brf-schedule:*`). This design update changes UI implementation approach and interaction primitives only; no functional behavior, stories, or specifications are changed.

The main application navigation is now tabbed. Message templates are rendered inside the Message Templates tab content area, with a warm professional visual system implemented through Tailwind CSS v4 and Bits UI primitives.

== Decisions and Constraints

* <<../../adrs/0001-svelte5-vite6-typescript-static-site.adoc#,ADR-0001>>: Svelte 5 with runes, Vite 6, TypeScript.
* <<../../adrs/0002-browser-localstorage-persistence.adoc#,ADR-0002>>: Browser localStorage with `brf-schedule:*` namespace.
* <<../../adrs/0003-tailwindcss-v4-bits-ui-warm-tabbed-redesign.adoc#,ADR-0003>>: Tailwind CSS v4 + Bits UI + tabbed UI redesign.

From `docs/constraints.adoc`:

* Static site deployable to GitHub Pages.
* localStorage-only persistence with `brf-schedule:` key prefix.
* Swedish user-facing text.
* TypeScript strict mode and test coverage non-regression.
* Minimized dependency footprint; use popular packages and keep runtime dependency overhead out of production dependency declarations.

No new ADR is required for this feature update. ADR-0003 already governs the migration to Tailwind CSS v4, Bits UI primitives, and tabbed navigation.

== Research and Unknowns

No open unknowns remain.

Resolved findings:

* Existing message template domain model and rendering logic are reusable without behavior changes.
* Existing storage contract (`brf-schedule:templates`) remains unchanged.
* Existing spec IDs (TMPL-001..TMPL-032) can be preserved with UI component replacement.
* Delete confirmation moves from browser `confirm()` to Bits UI `Dialog` while preserving confirmation semantics.

=== UI Composition and Styling Approach

The message template UI is implemented as a sectioned tab surface within the app's tabbed shell:

* Top-level app tabs: Building, Schedule, Message Templates, Print.
* Inside Message Templates: section tabs for editor-focused and preview-focused workflows.
* Tailwind CSS v4 utilities define spacing, typography, color tokens, panel backgrounds, borders, and responsive layout.
* Bits UI `Tabs` provide app-level and feature-level sectioning.
* Bits UI `Select` powers placeholder field mapping and apartment selection.
* Bits UI `Button` is used for create/save/delete/add actions.
* Bits UI `Dialog` handles delete confirmation.
* Bits UI `Label` provides accessible field labeling.

Layout behavior:

* Desktop: two-column working layout (template list + active editor/preview content).
* Mobile: stacked sections with preserved action order and full functionality.
* The immediate-editor behavior on new template creation is preserved via selected template state update before render.

=== Data Model

Data model remains unchanged from prior design version.

All types remain in `src/lib/models/template.ts`:

* `MessageTemplate`
* `PlaceholderMapping`
* `DataField` (`apartmentId`, `floor`, `position`, `date`, `time`, `dateSwedish`, `dateFormatted`)

No schema or persistence format changes are introduced.

=== Business Logic

Business logic in `src/lib/models/template-model.ts` remains behaviorally unchanged:

* Placeholder parsing with `{placeholder_name}` syntax.
* Placeholder-to-field mapping.
* Template rendering with apartment and appointment data.
* Unmapped placeholder detection for preview error presentation.
* ISO date/time handling and locale-aware date formatting with Swedish fallback.
* Backward compatibility for templates already using legacy `dateSwedish` mapping.

=== Persistence

Persistence in `src/lib/services/storage.ts` remains unchanged:

* key: `brf-schedule:templates`
* `saveTemplates()`
* `loadTemplates()`
* `clearTemplates()`

The UI migration does not alter storage keys, serialized structure, or load/save timing expectations.

== Story Resolution

=== US-001: Create Message Template

Components:: App Tabs (Bits `Tabs`), MessageTemplatePanel, TemplateEditor, Bits `Button`, Bits `Label`, template-model
Criteria::
* Text editor for composing -> TemplateEditor renders body editor area within tab content (TMPL-001)
* Template name input -> TemplateEditor uses labeled name input (`Label`) (TMPL-002)
* Persist on save -> Save action triggers `saveTemplates()` persistence (TMPL-003)
* Immediate editor display on creation -> New-template button sets selected template and renders editor immediately (TMPL-024)
* Multiple saved templates -> Templates are stored and displayed as a collection (TMPL-004)

=== US-002: Insert Placeholders in Template

Components:: TemplateEditor, Bits `Select`, Bits `Button`, Bits `Label`, template-model
Criteria::
* `{placeholder_name}` syntax -> `parsePlaceholders()` recognizes token syntax (TMPL-005)
* Create custom placeholder -> Add-placeholder flow with labeled input and action button (TMPL-006)
* Map to data field -> Each placeholder maps via Bits `Select` to a supported field (TMPL-007)
* Display available fields -> Field options are shown from `getAvailableDataFields()` (TMPL-008)
* Visually distinct in preview -> Rendered preview distinguishes unresolved placeholder states (TMPL-009)

=== US-003: Edit Existing Template

Components:: MessageTemplatePanel, TemplateList, TemplateEditor, Bits `Button`, Bits `Dialog`, storage
Criteria::
* List of saved templates -> TemplateList presents all saved templates (TMPL-010)
* Load into editor on select -> Selecting a template updates active editor content (TMPL-011)
* Update stored template -> Save persists modified template data (TMPL-012)
* Delete action -> Each template row exposes delete action button (TMPL-013)
* Confirmation before delete -> Delete flow requires explicit confirmation in `Dialog` (TMPL-014)

=== US-004: Preview Rendered Template

Components:: TemplatePreview, Bits `Select`, template-model
Criteria::
* Preview mode with actual data -> Preview panel renders resolved output for selected apartment (TMPL-015)
* Apartment selector -> Preview uses Bits `Select` for apartment choice (TMPL-016)
* Replace placeholders -> `renderTemplate()` resolves mapped placeholders (TMPL-017)
* Error style for unmapped -> Unmapped placeholders are rendered in explicit error style (TMPL-018)

=== US-005: Default Placeholders

Components:: MessageTemplatePanel, TemplateEditor, Bits `Button`, template-model
Criteria::
* Suggested placeholders on new template -> Default placeholders are provided at template creation (TMPL-019)
* Pre-mapped to fields -> Defaults remain pre-mapped to existing fields (TMPL-020)
* Remove or modify suggested -> Placeholder entries are removable/editable in editor UI (TMPL-021)

=== US-006: Save Templates

Components:: MessageTemplatePanel, storage
Criteria::
* Persist on create/modify -> Template state changes are persisted to storage (TMPL-003)
* Restore on load -> Existing templates restore when app loads (TMPL-022)
* Empty list when no data -> Empty-state list behavior is preserved (TMPL-023)

=== US-007: Locale-Aware Data Field Formatting

Components:: template-model, schedule-model, TemplateEditor, TemplatePreview, Bits `Select`
Criteria::
* Locale-formatted date rendering -> `dateFormatted` resolves via browser-locale Intl formatting (TMPL-025)
* Locale-formatted date field option -> Mapping UI includes locale-formatted date option (TMPL-026)
* Backward compatibility with dateSwedish -> Legacy `dateSwedish` mappings remain resolvable (TMPL-027)
* Swedish fallback -> Locale-resolution failure falls back to Swedish formatting (TMPL-028)

=== US-008: ISO Date/Time Placeholders

Components:: template-model, TemplateEditor, TemplatePreview, Bits `Select`
Criteria::
* ISO date placeholder option -> Mapping UI includes ISO date option (TMPL-029)
* 24-hour time placeholder option -> Mapping UI includes HH:MM time option (TMPL-030)
* New template default `{datum}` mapping -> Default mapping remains ISO date for new templates (TMPL-031)
* Backward compatibility -> Legacy Swedish date mappings continue to resolve (TMPL-032)
